# DexClub MCP V3 运行时可观测性

## 目标

本稿只定义 `mcp` 运行期的可观测性边界，回答以下问题：

1. 终端默认应输出哪些最小日志
2. 哪些高频轨迹应进入详细日志，而不是刷到终端
3. JVM / native 崩溃证据应如何保留
4. 运行时日志开关的职责边界应如何划分

本稿不重复讨论：

- MCP tool 面与字段契约
- skill 编排规则
- release / packaging 结构

这些内容见：

- `43-mcp-tool-surface.md`
- `44-skill-workflow.md`

## 设计原则

`mcp` 是一个长期运行的工具服务，不是交互式 CLI。

因此运行期日志策略不应走两个极端：

- 终端完全静默，导致出错时黑盒
- 所有请求轨迹都刷到终端，导致正常使用时噪音过大

更合理的分层是：

- 终端：最小运行日志
- 文件：详细轨迹日志
- JVM / native：崩溃文件

一句话说：

- 终端负责“当前是否正常”
- 详细日志负责“出了问题如何回看”
- 崩溃文件负责“进程为什么直接死”

## 终端最小日志

终端默认应始终保留以下最小日志：

### 1. 启动成功

至少输出：

- 监听地址
- 当前是否开启详细轨迹
- crash 文件默认写入位置

例如：

- `DexClub MCP listening on http://127.0.0.1:8787/mcp ...`

### 2. tool failure

当某个 tool 执行失败时，终端应直接可见：

- tool 名
- 错误类型
- 错误摘要

这里不应要求用户先去翻文件。

`tool failure` 同时也应进入详细日志文件，以便保留完整排障上下文。

### 3. 未捕获异常

若运行时出现未捕获异常，终端应直接输出：

- thread
- 当前 operation
- exception class
- message

### 4. 进程退出

当进程关闭时，终端应至少输出：

- shutdown 事件
- 当前 operation（若有）

这样可以区分：

- 人工 `Ctrl+C`
- 正常停止
- 某次处理中退出

## 详细日志文件

以下高频轨迹不应默认刷到终端，而应进入详细日志文件：

- HTTP request
- HTTP response
- HTTP failure
- MCP tool begin
- MCP tool end
- 参数摘要
- 请求耗时

这些日志的价值是：

- 事后排障
- 对齐 agent 行为
- 判断问题出在握手、tool 调用还是底层执行

但它们的缺点也很明显：

- 数量多
- 正常运行时噪音大
- 容易淹没真正异常

因此更适合写入文件，而不是长期占据终端。

## 详细日志文件位置与命名

当前活跃日志文件应保持固定路径：

```text
logs/mcp.log
```

这里的相对基准是分发启动脚本所在目录；若不是通过分发脚本启动，则退回当前工作目录。

它的含义是：

- `logs/mcp.log`
  - 当前 MCP 进程的活跃写入文件
  - 不承载上一次进程遗留内容

每次 MCP 进程启动时，如果发现旧的 `logs/mcp.log` 已存在且非空，应先归档到：

```text
logs/archive/mcp_<启动时间>.log
```

例如：

```text
logs/archive/mcp_20260522_204155.log
```

若同一秒内发生文件名冲突，可以追加数字后缀：

```text
logs/archive/mcp_20260522_204155_1.log
```

这样做的原因是：

- 当前存活进程的日志路径固定，用户容易找到
- 旧进程日志不会混入当前排障视图
- 归档文件带启动时间，便于对应 MCP 重启事件
- `archive/` 目录避免历史日志挤占主日志目录视线

## 详细日志文件归档策略

当前阶段不做运行中的大小轮转。

推荐语义是：

- `mcp.log` 始终是当前进程日志
- 历史日志只在下一次启动时归档
- 归档时间采用“本次启动时间”，而不是旧文件最后写入时间
- 默认只保留最近 10 份归档，超出后删除最旧文件
- `DEXCLUB_MCP_TRACE=false` 时，不创建详细轨迹文件，也不执行详细轨迹归档

不做运行中轮转的原因是：

- MCP trace 当前只记录 HTTP/tool 摘要，不写完整导出正文
- 单次进程日志连续保留，更便于按时间线排障
- 避免 `mcp.log.1` 同时承载“当前进程早期日志”和“上一进程日志”的歧义
- 日志策略保持轻量，不引入复杂 sink 或 logger 配置

这个策略的代价是：

- 如果单个 MCP 进程长期运行且 trace 量很大，`mcp.log` 会持续增长
- 归档保留数量可通过 `DEXCLUB_MCP_MAX_TRACE_ARCHIVES` 调整

## 事件边界

详细日志写入仍应保持事件完整性：

这里的“完整日志事件”包括：

- 单条普通日志
- 一整段异常堆栈

这样可以保证：

- 日志边界可读
- 排障时不需要拼接半条异常
- archive 中的旧日志也是完整文件级归档

## 日志文件内容范围

详细日志文件应优先覆盖以下字段：

### HTTP 层

- method
- uri
- json-rpc id
- rpcMethod
- tool（若为 `tools/call`）
- contentType
- accept
- session 相关标识
- elapsedMs

### tool 层

- tool begin
- tool end
- tool failure
- 当前关键参数摘要

参数摘要应保持克制：

- 只记录高价值键
- 避免无上限记录全文本参数
- 避免把大文本导出结果本身写入日志

## crash 文件策略

普通日志无法覆盖 JVM / native 直接崩溃。

因此 `mcp` 运行期还应保留：

- `hs_err_pid*.log`
- `*.hprof`（例如 OOM）

通过分发脚本启动时，这些文件默认也应落在启动脚本同级目录；若不是通过分发脚本启动，则退回当前工作目录。

这些文件的职责是：

- 当进程没有机会打印 `failure` 日志时，仍保留致命崩溃证据

它们不属于终端日志，也不应混入常规 MCP 详细轨迹文件。

## 默认行为

推荐的默认行为是：

### 终端

- 最小运行日志：默认开启

### 文件

- 详细日志文件：默认开启，可通过显式开关关闭
- 开启后写入固定位置：
  - `logs/mcp.log`
  - 路径基准是分发启动脚本所在目录
- 每次启动先归档旧日志：
  - `logs/archive/mcp_<启动时间>.log`
- 默认保留最近 10 份归档
- 运行中不做大小轮转

### crash 文件

- 默认开启

这里的关键不是“默认越多越好”，而是：

- 正常使用时不过度打扰
- 出问题时又不至于完全黑盒

## 开关语义

运行期日志开关不应只叫 `DEBUG`，因为它承载的并不只是调试模式，而是：

- 详细轨迹记录
- tool 调用可见性
- 参数摘要输出

因此更合理的语义是：

- 终端最小日志：常开
- 详细日志文件：默认开启，可显式关闭

如果沿用环境变量，则其职责应表达为：

- 是否关闭高频详细轨迹文件
- 而不是是否“允许错误被看见”

错误、未捕获异常与 shutdown 不应依赖详细日志开关。

## 不建议的方向

当前不建议：

### 1. 默认把所有 HTTP / tool 轨迹都打到终端

这样会：

- 噪音过高
- 掩盖真正异常
- 降低日常使用可读性

### 2. 完全关闭终端异常信息，只写文件

这样会：

- 让用户误以为服务无响应
- 出错时仍然像黑盒

### 3. 为日志策略引入过重的抽象

例如一开始就做：

- 多文件轮转
- 多级 logger 体系
- 复杂 sink 配置

当前阶段更重要的是：

- 先把最小层级和职责边界定清楚

## 一句话结论

DexClub MCP 的运行时可观测性应采用：

- `终端最小日志 + 详细日志文件 + JVM/native crash 文件`

而不是：

- 完全静默
- 或终端全量刷屏
