# DexClub V4 定位

## 目标

本稿只回答四件事：

- `v4` 是什么
- `cli / mcp / gui` 各自是什么
- 共享能力边界应该放在哪里
- 哪些东西不应该被入口层占住

## 定位

`DexClub V4` 是面向 dex / apk / Android resources 静态分析的共享能力内核，以及建立在其上的多入口分析产品。

当前实现范围包括：

- `cli`
- `mcp`

未来兼容范围包括：

- `gui`

直接结论是：

- `cli` 只是命令行入口
- `mcp` 只是工具协议入口
- `gui` 只作为未来兼容入口，不作为当前实现范围
- 共享能力必须先围绕长期入口共性收敛，而不是继续围绕单一入口扩张

## 共享边界

共享能力应围绕“分析能力”和“稳定领域规则”收敛，而不是围绕具体入口的表现形式收敛。

共享能力应承接：

- workspace / target 的正式模型
- dex / resource 分析请求与结果模型
- 能力校验与执行前置规则
- 导出、查询、检查、解析等稳定用例
- 上下文、缓存、session、task 的正式生命周期边界

共享能力不应承接：

- CLI flags
- MCP schema 细节
- 文本渲染格式
- GUI 状态

## 入口边界

### CLI

`cli` 负责：

- 参数解析
- 命令路由
- 输出渲染
- 退出码

`cli` 不负责：

- 领域规则定义
- 会话与上下文生命周期
- 缓存策略
- 长任务调度

### MCP

`mcp` 负责：

- tool schema 暴露
- tool 调用参数解析
- 面向 agent 的结构化结果投影
- 协议层错误表达

`mcp` 不负责：

- 重新定义分析能力
- 长期持有业务规则副本
- 复制一套应用运行时

### GUI

`gui` 不属于当前实现范围。

但未来如果接入，它应复用同一套应用层，而不是自行再长出一层桌面专用运行时。

## 非目标

`v4` 不把下面这些内容作为先决目标：

- 先做一个完整 GUI 再回头整理架构
- 为兼容现有入口形态而保留明显不合理的内部边界
- 先把所有模块改名，再讨论职责
- 在没有稳定应用层前继续大幅扩张命令面或 tool 面

## 设计原则

- 共享能力优先围绕长期入口共性设计
- 入口层只做适配，不持有领域主逻辑
- 长生命周期状态必须有正式归属，不能分散挂在各入口
- GUI 兼容约束应尽早进入设计，但 GUI 本身不属于当前实现范围
- 模块名字不是重点，职责边界才是重点

## 当前结构口径

当前代码结构直接按下面口径理解即可：

- 根工程子项目
  - `domain-core`
    - 稳定模型、能力边界与底层实现
  - `app-service`
    - 应用层
  - `cli-app`
    - 命令行入口
  - `mcp-app`
    - MCP 工具入口
- 独立组合构建
  - `dexkit-binding`
    - 绑定层

未来如果要做模块重整，也应围绕这组职责推进，而不是先追求一套新的模块命名。
