# DexClub V4 GUI 兼容边界

## 目标

本稿不讨论 GUI 实现方案。

本稿只回答一个问题：

- 在 `v4` 当前不实现 GUI 的前提下，结构上至少要避免哪些问题

## 当前范围

GUI 不属于 `v4` 当前实现范围。

`v4` 当前对 GUI 的讨论只保留在兼容边界层面：

- 未来如果增加 GUI，现有结构不应成为明显阻碍
- 当前不为了 GUI 提前写一整套桌面实现
- 当前不展开视图、交互、界面状态和桌面技术选型

## 需要提前避免的问题

即便现在不实现 GUI，结构上也应避免下面几类问题：

### 1. 把运行时职责锁死在 `cli`

如果 workspace 恢复、上下文拼装、缓存复用、错误语义都长期留在 `cli`，那么未来 GUI 只能复制一遍。

### 2. 把会话职责锁死在 `mcp`

如果 session、handle、dex context 生命周期被默认视为 `mcp` 私有逻辑，那么未来 GUI 需要类似能力时就只能绕路。

### 3. 默认所有操作都是瞬时同步调用

GUI 虽然当前不实现，但未来一旦接入，就天然更依赖：

- 渐进加载
- 取消
- 后台任务
- 上下文复用

所以应用层接口不应被设计成彻底排斥这些能力。

### 4. 让共享能力依赖入口表现形式

共享内核不应依赖：

- 命令行参数形态
- tool schema
- 文本渲染格式
- 组件状态

否则未来 GUI 接入时只能穿透现有适配层。

## 当前结论

在 `v4` 当前范围内，GUI 相关结论先收敛到这几条：

- 不实现 GUI
- 不讨论 GUI 细节设计
- 但要把应用层补出来
- 让 `cli` 和 `mcp` 变薄
- 避免未来 GUI 需要复制 `cli` 或 `mcp` 的运行时逻辑

在那之前，`v4` 的重点仍然是：

- `application-layer.md`
- `adapters.md`
- `workspace-and-session.md`
- `migration.md`
