# DexClub V4 迁移路径

> 本稿是 `v4` 的迁移背景文档。
> 它主要回答“这轮结构为什么按这个顺序推进”，不是当前仓库的逐项施工清单。
> 当前是否完成、还差什么，应优先看 [completion.md](./completion.md) 和 [stabilization.md](./stabilization.md)。

## 目标

本稿只回答下面几个问题：

- 从早期实现推进到 `v4`，顺序当时为什么这样排
- 哪些事情应该先做
- 哪些事情不应该一开始就做
- 每个阶段的完成标准是什么

这是一份迁移背景文档，不是当前主线入口；当前重点见 [stabilization.md](./stabilization.md)。

## 总原则

当时推进 `v4` 时，不应把“迁移”理解成：

- 先改完所有模块名
- 先新建一堆目录
- 先把现有代码整体推翻

更合理的理解是：

- 先把共享运行时边界收口
- 再把入口层变薄
- 再整理模块结构
- 最后再处理命名、构建链和交付形态

也就是说：

- 先改职责
- 再改结构
- 最后改外观

## 不应优先做的事

迁移初期不建议优先做下面这些动作：

- 大规模改模块名
- 大规模移动文件
- 为了追求目录“好看”先拆出很多空模块
- 先把 GUI 模块搭起来
- 先把 README 改成 `v4` 最终形态
- 在应用层没定之前继续扩张命令面和 tool 面

这些事情不是永远不做，而是不应排在最前面。

## 建议迁移阶段

这条迁移路径仍可按六个阶段理解，但今天再看时要注意：

- 这六个阶段主要用于解释推进顺序
- 它们不再等于当前仓库的逐条待办
- 其中前几阶段已经大幅落地，后几阶段也已有明确结果映射

## 阶段 0：冻结边界漂移

### 目标

在进入迁移前，先避免继续增加结构负债。

### 需要做到

- `cli` 不再新增共享运行时逻辑
- `mcp` 不再新增非协议层主规则
- 新能力实现优先放在现有共享层，而不是入口层
- 新文档先以 `.docs/v4/` 为准继续收敛

### 完成标准

- 新增代码不再继续把 session / cache / context 规则分散进入口
- 后续设计讨论以 `v4` 文档为主

## 阶段 1：收口应用层边界

### 目标

先把“应用层到底负责什么”从文档变成代码中的明确边界。

### 需要做到

- 在现有结构中抽出正式的应用层入口
- 统一承接 workspace / session / handle / dex context 相关运行时职责
- 让入口层开始通过应用层调用，而不是直接拼底层 service

### 这阶段先不要做

- 不要求立刻改模块名
- 不要求立刻移动所有实现
- 不要求立刻消灭现有领域底座里的全部默认装配

### 推荐动作

- 先新增应用层 package 或模块骨架
- 先把最清晰的一批用例迁进去
- 先把 `mcp` 中最明显的 session / handle 规则往应用层收

### 完成标准

- 代码里已经存在可识别的应用层入口
- 至少一条完整业务链路通过应用层贯通
- `cli` 和 `mcp` 都开始依赖这条新边界

### 当前状态

这一步对 `mcp` 来说已经基本成立：

- `app-service/dex`
- `app-service/resource`
- `app-service/session`

都已经开始承接正式 use case 或上下文恢复逻辑。

## 阶段 2：先迁 `mcp` 运行时，再迁 `cli`

### 目标

优先从 `mcp` 回收运行时职责。

### 为什么先动 `mcp`

因为当前最明显的运行时概念集中在 `mcp`：

- session
- handle
- dex context lease
- 一部分上下文恢复

这些职责天然更接近应用层，而不是协议层。

### 推荐动作

- 先把 target session 的主生命周期规则收回应用层
- 再把 handle 的主生命周期规则收回应用层
- 再把 dex context 的主持有与释放规则收回应用层

### 对 `cli` 的要求

这个阶段不要求 `cli` 大改，但应开始配合：

- 减少自己做前置执行拼装
- 改为调用应用层用例

### 完成标准

- `mcp` 不再持有主要的 session / handle / dex context 规则
- `mcp` 更接近纯协议适配层

### 当前状态

这一步当前处于“主链路已收口，但桥接仍在”的状态：

- dex / resource 主执行语义已明显下沉
- 请求解析、结果映射、协议错误包装已稳定留在 `mcp`
- `mcp` 侧的过渡命名已经去掉，但上下文恢复与 lease 获取已明确收进 `app-service/session`

## 阶段 3：重塑用例接口

### 目标

把“很多底层 service 方法”收口成“少量正式用例入口”。

### 需要做到

- 把高频链路整理成稳定 use case
- 让入口层不再自己组合多个共享 service 才能完成一次正常调用
- 统一 request / result 模型

### 推荐先迁的链路

- workspace 打开 / 初始化 / refresh
- dex 查询
- inspect method
- export
- resource 查询与解析

### 完成标准

- 入口层调用开始围绕 use case，而不是围绕底层实现分类
- 共享错误语义和上下文恢复逻辑不再散落在入口中

### 当前状态

这一步对 `mcp` 也已经明显推进：

- `inspect / find / export / resource` 等高频链路已经围绕 use case 组织
- tool 层主要做参数读取、结果投影和协议包装

对 `cli` 来说，这一步当前处于“按链路分批接入”的状态：

- `inspect-method`、一批 resource 链路与整组 dex 查询链路已经开始复用 `app-service` use case
- 这些已接入的 use case 装配已经开始集中到 `app-service/AppUseCases`
- 导出、`inspect`、`res-table` 与 workspace 命令组也已经开始复用 `app-service` use case
- 当前 `cli` 已没有直接直连 `services.*` 的命令适配链路
- 这一步的风险控制已经从“分批迁移”进入“收口后的稳定校验”

## 阶段 4：整理模块边界

### 目标

在职责已经稳定之后，再做模块和目录重组。

### 需要做到

- 根据已经落地的职责，决定是否真的需要正式模块重整
- 如果需要，再判断哪些边界值得独立成模块
- 清理已经失去价值的旧边界

### 这阶段为什么不能提前做

如果在职责还不稳定时先拆模块，结果通常是：

- 名字改了
- 目录变了
- 共享逻辑仍然散落

那只是结构换皮，不是迁移完成。

### 完成标准

- 模块边界开始对应真实职责，而不是历史遗留目录
- 依赖方向更接近 `architecture.md` 的目标结构

## 阶段 5：收口构建与交付

### 目标

在运行时边界和模块边界稳定后，再整理构建链和交付形态。

### 需要做到

- 明确 native / vendor / 本地构建链的正式位置
- 决定哪些是主仓直接维护内容
- 决定哪些应转成更清晰的产物消费关系
- 收口 README 与最终开发说明

### 完成标准

- 构建链说明与实际结构一致
- 开发者能区分共享内核变更和入口层变更
- native 相关前提不会继续泄漏到所有日常改动路径

## 哪些链路适合作为第一批迁移对象

优先级较高的候选是：

### 1. `mcp` session 链路

原因：

- 运行时职责最集中
- 回收价值最大
- 边界最容易说明白

### 2. `inspect method` 链路

原因：

- 跨越 workspace、dex 查询、结果映射
- 能很好暴露入口层和共享层之间的边界问题

### 3. `find methods` 或 `find classes` 链路

原因：

- 高频
- 结果模型明确
- 适合作为 use case 收口样板

## 哪些链路不建议作为第一批

当前不建议第一批就从下面这些方向切：

- 先全面重写全部 CLI parser
- 先全面重做输出模型
- 先动所有 resource 链路
- 先处理 GUI 兼容层以外的设计
- 先做 native / vendor 构建大重排

这些事情都可以做，但放在前面会让迁移目标失焦。

## 每阶段的验证建议

### 阶段 1 到阶段 3

重点验证：

- `:app-service:testStructured`
- `:domain-core:testWorkspace`
- `:cli-app:testStructured`
- `:mcp-app:testStructured`

尤其要关注：

- 行为是否保持一致
- 入口输出契约是否被意外改变
- 生命周期规则是否只换位置、不改语义

### 阶段 4 到阶段 5

重点验证：

- 模块依赖是否仍然清晰
- 构建任务是否仍能正常运行
- 文档是否和新边界一致

## 迁移顺序示意

如果只想理解“为什么当时这样推进”，可按下面顺序理解：

1. 先完成应用层代码入口
2. 把剩余 `session store / dex context registry` 所有权继续定清
3. 再让 `cli` 更全面地收口到同一批 use case
4. 再评估是否值得正式模块整理
5. 最后再做构建链收口

当前第 3 步更准确地可以拆成：

- 先迁参数与现有 use case 明显对齐的 `cli` 链路
- 再补齐查询/导出侧仍缺的应用层入口
- 最后再决定是否需要继续把剩余 adapter 全量切走

## 迁移完成的判断标准

只有满足下面这些条件，才算真正接近 `v4`：

- `cli` 主要只做命令适配
- `mcp` 主要只做协议适配
- 应用层已经正式持有共享运行时职责
- `workspace / session / handle / dex context` 的关系在代码里有清晰归属
- 模块或包边界开始对应真实职责，而不是历史目录

这里要特别区分两件事：

- `v4` 主文档已经齐备
- `v4` 实现已经完成

前者只说明讨论入口和主边界已经成形。

后者至少还要求：

- `cli` 不再大面积直连 `services.*`
- `cli` 命令适配层不再直接拼 `services.*`
- `mcp` 只保留合理的协议层桥接
- 是否做正式模块重整已经有明确结论，并且文档与代码状态一致
- 构建、交付与文档已经随新结构一起收口

如果这些条件还不成立，就不能把当前状态描述成“收尾”或“接近完成”。

## 当前状态

这份文档现在更适合按“迁移背景”阅读，而不是按“当前待执行阶段”阅读。

当前仓库的活跃模块结构已经落到：

- `cli-app`
- `mcp-app`
- `app-service`
- `domain-core`
- `dexkit-binding`

因此这里保留的阶段描述，主要用于解释：

- 为什么当时先收口应用层边界
- 为什么入口层改名放在后面
- 为什么 `dexkit-binding` 最后落地

如果要判断现在 `v4` 还差什么，不应再回到这份阶段文档里找迁移任务，而应直接看：

- [completion.md](./completion.md)
- [stabilization.md](./stabilization.md)
