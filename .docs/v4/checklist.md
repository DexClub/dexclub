# DexClub V4 边界检查清单

> 本稿是当前边界检查单。
> 它用于判断“这次改动有没有把职责重新散回去”，不是迁移阶段说明，也不是新的结构改造清单。
> 如果需要理解推进顺序，优先看 [migration.md](./migration.md)；如果需要看当前稳定边界，优先看 [stabilization.md](./stabilization.md)。

## 目标

本稿用于在边界相关改动后反复检查一件事：

`这次改动是不是仍然在维持 v4 的稳定边界，而不是把职责重新散回入口层`

## 使用方式

每一轮较大的实现、重构、入口调整或模块调整之后，都建议至少过一遍这份清单。

它更适合在下面这些场景使用：

- 完成一条 use case 迁移后
- 调整 `cli-app / mcp-app / app-service` 边界后
- 更新构建与交付说明后
- 做架构相关 code review 时
- 判断一轮重构是否真的改善边界时

这里默认站在“当前代码已经是 `v4` 活跃结构”的前提上检查：

- 根工程子项目：`app-service / domain-core / cli-app / mcp-app`
- 独立绑定工程：`dexkit-binding`

## 一、应用层

### 1. 是否真的在收口应用层

确认：

- 共享运行时职责正在进入应用层
- 新增用例优先通过应用层组织
- 入口层不再继续直接拼底层 service

如果新增逻辑仍然直接长在 `cli` 或 `mcp`，应先解释为什么不能进应用层。

### 2. 应用层是否承担了正确职责

确认应用层承接的是下面这些内容：

- 用例入口
- 执行上下文恢复
- workspace / session / handle / dex context 生命周期
- 共享缓存规则
- 共享错误语义

确认应用层没有承接下面这些内容：

- CLI 参数解析
- MCP schema 细节
- 文本渲染
- HTTP 协议

### 3. 应用层是否只是换皮 facade

检查是否只是新增了一个名字更大的壳，但实际上：

- 仍然没有统一上下文恢复
- 仍然没有统一生命周期规则
- 仍然只是把底层 service 原样转调出去

如果是，就不算完成应用层收口。

## 二、入口适配层

### 4. `cli` 是否继续变薄

确认 `cli` 主要只负责：

- argv 解析
- 命令路由
- query file 读取
- 输出渲染
- 退出码

确认 `cli` 没有继续新增：

- session 主规则
- dex context 生命周期规则
- 共享缓存策略
- 共享错误分类规则

如果当前改动仍在 `cli` 内部，进一步确认：

- 是否优先复用了现成 `app-service` use case
- 如果还保留直连 `services.*`，是否是因为对应 use case 尚未稳定，而不是偷懒绕开应用层

### 5. `mcp` 是否继续变薄

确认 `mcp` 主要只负责：

- tool schema
- 参数解析
- 结果投影
- 协议层错误表达

如果当前改动仍在 `mcp` 内部，进一步确认它是否属于下面这些合理残留：

- 请求解析 support
- 字段裁剪与结果映射
- 协议层诊断与日志
- 过渡期宿主态桥接

确认 `mcp` 没有继续新增：

- session 主生命周期规则
- handle 主生命周期规则
- dex context 主持有与释放规则
- 与协议无关的共享执行逻辑

如果本轮改动继续增厚 session service 或 dex context registry 的入口桥接，应先说明为什么这不是把过渡桥接重新做重。

### 6. 入口层是否仍然在复制共享逻辑

检查：

- 同一条前置校验是否在多个入口重复出现
- 同一条上下文恢复规则是否在多个入口重复出现
- 同一条生命周期规则是否在多个入口重复出现

如果答案是“是”，应继续回收到应用层。

## 三、核心概念边界

### 7. `workspace` 和 `session` 是否被区分清楚

确认：

- `workspace` 仍是正式执行上下文
- `session` 仍是运行期会话概念
- 两者没有重新混成一个对象

### 8. `handle` 是否被错误上升为领域对象

确认：

- `handle` 没有进入核心领域模型
- `handle` 仍然依赖会话或运行期作用域
- 不是所有入口都被迫显式理解 `handle`

### 9. `dex context` 是否仍被当成运行时资源

确认：

- `dex context` 没有泄漏成入口主模型
- 创建、复用、释放规则由应用层控制
- 绑定层没有反向持有会话主规则

## 四、用例接口

### 10. 用例接口是否在收口

确认：

- 入口调用开始围绕 use case，而不是围绕底层实现分类
- 一次正常操作不需要入口自己组合多个共享 service
- request / result 模型开始稳定

如果是 `mcp` 改动，进一步确认：

- tool 实现是否主要只剩参数读取、use case 调用、结果投影

### 11. 是否仍在扩张“很多 service 方法”的旧形态

检查新增能力时，是否仍然优先走下面这种路径：

- 先给底层 service 再加一个方法
- 再让每个入口自己适配一次

如果是，应优先判断是否应该直接落成正式 use case。

如果当前是 `cli` 改动，也要避免另一种退化：

- 明明已有对应 use case，却继续在 adapter 里直接拼 `services.dex / services.resource / services.workspace`

## 五、构建与交付

### 12. 共享层开发路径是否保持轻量

确认：

- 只改共享层时，不会被无意义地要求准备完整 native 维护环境
- 文档没有把维护链前提扩散成所有开发路径的默认要求

### 13. native / vendor 维护路径是否被清楚标记

确认：

- native / vendor 相关改动有独立说明
- 复杂环境前提被清楚标记为维护路径要求
- 主仓高层文档没有把它们伪装成所有场景都必需

### 14. 交付路径是否和维护路径分离

确认：

- CLI / MCP 打包路径说明清楚
- 入口交付与 native 重建不是同一件事
- 运行前提与维护前提被清楚区分

## 六、职责与结构

### 15. 是否遵守先职责后结构

确认：

- 先收口职责，再考虑结构
- 没有为了“目录整齐”提前做大规模移动
- 没有在应用层未稳定时先全面改名

### 16. 是否优先收口了高价值链路

当前更合理的优先级应大致符合：

- 先 `mcp` session / handle / dex context
- 再高频 dex 链路
- 再入口全面收口
- 最后模块整理和构建链整理

如果当前已经完成高频 dex / resource 链路 use case 化，则后续优先级应转向：

- 剩余运行时桥接的最终归属
- `cli` 是否同步收口到同一应用层边界

如果顺序完全反过来，应检查是否已经跑偏。

## 七、行为与验证

### 17. 行为是否保持一致

确认：

- 迁移主要是在换边界，不是在顺手改行为
- 若行为必须变化，已经明确记录原因和影响面

### 18. 输出契约是否被意外破坏

尤其检查：

- CLI text 输出
- CLI JSON 输出
- MCP result 结构
- 错误输出格式

### 19. 是否做了最小可用验证

至少确认本轮直接影响到的链路已经验证：

- `:app-service:testStructured`
- `:domain-core:testWorkspace`
- `:cli-app:testStructured`
- `:mcp-app:testStructured`

如果本轮改动涉及构建链，还要额外确认相关构建任务。

## 八、文档一致性

### 20. 文档是否仍与当前代码方向一致

确认：

- `.docs/v4/` 中的边界说明没有被代码悄悄绕开
- 若本轮改变了结论，文档已同步更新
- README 没有抢占 `.docs/v4/` 的设计职责

## 通过标准

一轮实现只有在下面条件大致同时成立时，才算真正符合 `v4` 的边界要求：

- 应用层更清楚了
- 入口层更薄了
- 共享运行时职责更集中了
- 概念边界更稳定了
- 构建和交付边界更清楚了
- 文档和代码没有重新分叉
